    3.21. Введение в вишнёвый сбор

    Команду git cherry-pick из-за дословного перевода называют вишнёвым сбором или снятием вишни. Эту команду часто совмещают с git merge и git rebase. То есть сначала вы снимаете вишню, потом делаете перебазирование, а заканчиваете всё слиянием. При такой комбинации сохраняется линейная история, но если вы, например, опустите git rebase, то линейной истории не будет.

    Что делает git cherry-pick

    Команда git cherry-pick создаёт точную копию коммита или группы коммитов.

    Далее для простоты мы будем говорить в единственном числе — упоминать только один коммит. Но помните, что команда может работать и с группой.

    В эту команду можно передать начальный и конечный коммит. Тогда копии будут созданы для переданных коммитов и тех, что находятся между ними.

    Посмотрите на рисунок ниже. У нас есть граф репозитория, и мы сейчас мы находимся на ветке develop. Мы хотим создать точную копию коммита 5e4d, который был создан после создания ветки develop.

    Команда git cherry-pick берёт коммит из переданной ветки (написанной после команды), копирует изменения, вставляет их в текущую ветку и создаёт новый коммит с таким же текстом. Хэш у такого коммита будет другим, несмотря на то, что вы сделали точную копию коммита. Это можно увидеть на рисунке ниже: у нас появился коммит с хэшем 68af, хотя на ветке main был коммит с хэшем 5e4d.

    Команда может принимать опцию --no-commit. Тогда после ввода команды изменения перенесутся в текущую ветку и сразу добавятся в индекс. Затем вы можете самостоятельно зафиксировать их.

    Получается, можно это делать самому, не используя команду? Да, вы можете скопировать изменения в коммите на одной ветке, затем их вставить в другую и после этого зафиксировать. Хэш коммита всё равно будет отличаться, даже если вы напишете такой же текст.

    У такого способа есть подводные камни — они связаны с комбинацией команд git cherry-pick и git rebase. Если вы будете самостоятельно создавать точную копию коммита, то хэш коммита будет другим, и коммиты не будут связаны между собой. Когда вы используете git rebase, Git посчитает скопированный коммит другим (так и есть) и добавит ещё коммит с таким же текстом, но другим хэшем к вам в ветку. В лучшем случае это вызовет путаницу: у двух коммитов одинаковый текст, а хэши разные. В худшем — приведёт к конфликту.

    При использовании git cherry-pick хэш коммита будет отличаться, но он будет содержать в себе особый механизм — идентификатор исправления (patch ID). Этот идентификатор сгенерируется, когда вы создадите точную копию коммита. Поэтому, когда вы будете использовать команду git rebase, Git определит, что этот коммит не так уж сильно отличается от основного. В результате Git просто заменит скопированный ранее коммит на новый входящий.

    По умолчанию Git сначала прочитает скопированный коммит, затем его удалит, а после вставит уже основной входящий коммит с таким же хэшем, что и на другой ветке. Ну, а если вы не будете использовать git rebase, то так как хэш коммита всё равно отличается, то при слиянии веток у вас будет два коммита с одним и тем же текстом. А ещё создастся коммит слияния. Ведь, скорее всего, в целевой ветке после появления дополнительной уже были созданы какие-то коммиты, а это значит, что слияние в режиме fast-forward недоступно.

    В каких случаях используют команду

    Хотя при использовании git cherry-pick скопированный коммит содержит идентификатор исправления, без команды git rebase от него мало толку: хэш коммита всё равно считается другим. Поэтому без git rebase круг использования команды сильно сужается — мало кому нужны дублирующие коммиты. И всё же есть несколько вариантов, где эта команда пригодится.

    Пример № 1. Представьте, что разработчики выпустили обновление, а через какое-то время выясняется, что в нём найден критический баг, позволяющий взломать приложение. Руководитель проекта просит разработчика срочно решить задачу. Разработчик в своей ветке вносит изменения, затем их фиксирует, переключается на основную ветку и создаёт точную копию коммита, сделанного на своей ветке. Потом это изменение отправляется в удалённый репозиторий.

    Почему бы не использовать git rebase или git merge? Дело в том, что у разработчика могут быть ещё какие-то коммиты на ветке, а эти команды перенесли бы все сделанные коммиты. Или другой вариант: кто-то до него исправил данную ошибку. Всё, что ему остаётся — скопировать коммит с той ветки, в которой он находится. А ещё, если он работает в одиночку или ему доверяют в компании, то, скорее всего, может сделать изменение сразу на основной ветке — тогда копировать ничего не придётся.

    Пример № 2. Восстановление удалённых коммитов.

    В этом варианте мы затронем команды, которые будем изучать позже. Поэтому просто воспринимайте текст как интересное чтиво.

    Допустим, один из разработчиков-новичков использовал команду git reset с опцией --hard и случайно полностью срезал коммит. Старший разработчик подсказал ему, что утерянный коммит можно вернуть с помощью команды git reflog, которая выводит журнал всех действий, а также команды git cherry-pick. То есть сначала вы узнаёте хэш утерянного коммита, а затем возвращаете его, используя git cherry-pick.

    Такое случается редко — большинство знает, что использовать git reset с опцией --hard нужно осторожно и обдуманно. Но всё же порой этот совет бывает полезен.

    Пример № 3. Самый распространённый вариант из предложенных нами. Допустим, вы с другом создаёте сайт и работаете на двух разных ветках. Во время работы вы понимаете, что вам срочно нужна часть кода, которую написал ваш друг. Сделать слияние или перебазирование он не может — у него есть другие коммиты. А вот сделать точечно копию одного коммита вы можете, что собственно и делаете.

    Отметим, что необязательно коммит переносить с тем же текстом. Вы можете перенести изменения без создания коммита, а когда доделаете свою часть, сделаете общий коммит — если изменения были в рамках одной задачи. Помним: делать коммит, охватывая сразу две разные задачи, — плохая практика.

    Вывод

    У git cherry-pick не так много вариантов использования. Применять её нужно осторожно и обдуманно, потому что она создаёт дубликат коммита. Если такой дубликат попадёт на ветку, в которой уже есть коммит с таким текстом, это может вызвать путаницу, история потеряет линейность. А если и git rebase не использовать, добавятся коммиты слияния — в истории будет ещё сложнее ориентироваться.
